## Design Context

### Users
GasStation is primarily a portfolio/reference Android app for reviewers and interviewers, with real driver usability kept intact as the product baseline. Reviewers should quickly see that the app has production-grade Android structure, coherent UI contracts, and a polished presentation layer. Drivers use the app to compare nearby gas stations by current location, price, distance, brand, fuel type, watchlist state, and external map handoff.

### Brand Personality
The interface should feel trustworthy, fast, and urban. The strongest visual association is a city public information sign: high-contrast, direct, and easy to scan under time pressure. The deeper craft goal is precise and production-grade rather than decorative.

### Aesthetic Direction
Keep the legacy yellow, black, and white identity, but refine it into a modern public-signage system. Price remains the first-read element on station cards. Distance, brand, watch state, freshness, and failure states support the price decision without competing with it. The app should avoid generic Material sameness, soft generic cards, and decorative effects that do not improve reading speed.

### Design Principles
1. Price is the hero, everything else is context.
2. Every screen should prove the same design system: tokens first, feature screens second.
3. Use bold contrast with disciplined rhythm, not visual noise.
4. Prefer fast scanning over ornamental polish.
5. Make review value visible: clear state handling, consistent components, accessible interaction targets, and predictable typography roles.

## Maintenance Priority

The current UI already implements the shared information hierarchy. Future UI work should preserve it before attempting a broader visual restyle:

1. Check `core:designsystem` tokens and shared chrome components before adding feature-local UI patterns.
2. Keep station card hierarchy ordered around price, distance, station name, brand, watch state, and delta.
3. Keep status, empty, loading, permission, GPS, and stale-cache states inside the shared guidance system.
4. Apply the same metric, row, supporting-info, and guidance rhythm to settings and watchlist while preserving their screen-specific contracts.
5. Use visual QA screenshots to confirm the README/demo story still reads well at portfolio-review speed after visible UI changes.
