# GasStation Portfolio Upgrade Design

## 목표

GasStation을 단순한 주변 주유소 조회 레퍼런스 앱이 아니라, 실무형 Android 설계 역량과 검증 습관을 함께 보여주는 포트폴리오 대표작으로 재포지셔닝한다.

이 설계의 핵심 목표는 아래 네 가지다.

- 위치, 권한, 오프라인, 외부 지도 연동이 섞인 실무형 탐색 앱이라는 점을 더 분명하게 드러낸다.
- 기존 멀티모듈 구조와 상태 분리 설계를 유지하면서도 제품 가치가 보이는 대표 기능을 추가한다.
- 테스트, 성능 측정, 분석 이벤트 설계를 통해 "구조를 설명하는 사람"이 아니라 "구조를 검증하는 사람"으로 보이게 만든다.
- README, 스크린샷, 시연 영상만 봐도 프로젝트의 강점이 짧은 시간 안에 이해되도록 패키징한다.

## 현재 상태 요약

현재 프로젝트는 이미 아래 강점을 갖고 있다.

- `app`, `feature`, `domain`, `data`, `core`, `benchmark`로 나뉜 멀티모듈 구조
- `demo` / `prod` flavor 분리
- Room 기반 마지막 성공 결과 유지와 stale 데이터 표시
- 외부 지도 앱 연동
- ViewModel, repository, datastore, database, location, domain 계층 전반에 걸친 단위 테스트 분포
- baseline profile 생성용 benchmark 모듈

반면 포트폴리오 관점에서 부족한 점은 아래 두 가지다.

- 면접관이 짧게 보고 기억할 "대표 장면"이 약하다.
- 테스트와 benchmark 기반은 있으나, 무엇을 왜 검증했는지 한눈에 읽히는 결과물로 연결되지 않았다.

## 추천 방향

확장 전략은 "대표 기능 1개 + 증거 3종 세트"로 잡는다.

- 대표 기능: 가격 변동 인사이트와 관심 주유소 비교
- 증거 3종 세트: 성능 측정, 테스트 신뢰성, 분석 이벤트 설계

이 방향을 선택하는 이유는 다음과 같다.

- 현재 앱의 핵심 가치인 "주변 탐색"을 유지한 채 한 단계 높은 의사결정 경험을 만들 수 있다.
- 기능 수를 과도하게 늘리지 않고도 제품적 사고를 보여줄 수 있다.
- 기존 아키텍처 강점을 그대로 살리면서 상태 모델, 데이터 저장 전략, 검증 범위를 더 설득력 있게 설명할 수 있다.

## 제품 설계

### 1. 리스트 화면 강화

기존 주유소 리스트 카드에 현재 가격과 거리 외에 가격 변화 배지를 추가한다.

예시 상태는 다음과 같다.

- `-20원 하락`
- `+35원 상승`
- `변동 없음`
- `비교 데이터 없음`

이 배지는 사용자가 "지금 어디가 싼가"뿐 아니라 "최근에 어떤 변화가 있었는가"를 빠르게 파악하게 해준다.

### 2. 관심 주유소 저장

사용자는 리스트 카드 또는 간단 상세 화면에서 주유소를 관심 항목으로 저장하거나 해제할 수 있다.

관심 저장 기능의 역할은 단순 즐겨찾기보다 넓다.

- 비교 화면의 입력값이 된다.
- 사용자의 명시적 의도를 영속 상태로 저장하는 예시가 된다.
- 제품 관점에서 반복 사용 시나리오를 만든다.

### 3. 관심 주유소 비교 화면

새 화면은 저장한 관심 주유소들을 비교 가능한 읽기 모델로 보여준다.

비교 항목은 아래로 제한한다.

- 현재 가격
- 마지막 확인 시각
- 최근 가격 변화
- 현재 위치 기준 거리

이 화면의 목적은 "가까운 곳"과 "싼 곳"이 다를 수 있다는 현실적인 판단 포인트를 드러내는 것이다. 범위를 넓히기 위해 복잡한 상세 화면이나 과도한 CRUD를 추가하지는 않는다.

### 4. 간단 상세 진입

풀스크린 상세 페이지 대신 바텀시트 또는 간단 상세 카드 수준의 경량 상세를 사용한다.

표시 항목은 아래로 제한한다.

- 브랜드
- 가격
- 거리
- 마지막 갱신 시각
- 관심 등록/해제
- 외부 지도 열기

이 결정은 기능 밀도는 높이되 구현 범위는 통제하기 위한 것이다.

## 아키텍처와 모듈 변화

기존 모듈 경계는 최대한 유지한다.

- `feature:station-list`
  리스트 카드의 가격 변화 배지, 관심 등록 액션, 비교 화면 진입점 추가
- 신규 `feature:watchlist`
  관심 주유소 비교 화면과 관련 UI 상태 담당
- `domain:station`
  관심 주유소 조회/토글, 가격 변화 계산, 비교용 읽기 모델에 필요한 계약과 유스케이스 추가
- `data:station`
  스냅샷 유지 외에 가격 히스토리와 관심 주유소 저장 책임 확장

초기 단계에서는 별도 도메인 모듈을 더 쪼개지 않는다. 범위를 과도하게 늘리는 대신, 현재 `domain:station` 안에서 읽기 모델과 유스케이스를 확장하는 편이 구현 효율과 설명력 모두에서 낫다.

## 데이터 모델 설계

현재 스냅샷 캐시는 유지하고, 아래 두 저장 단위를 추가한다.

- `StationPriceHistory`
  최근 가격 비교용 히스토리
- `WatchedStation`
  사용자가 저장한 관심 주유소 목록

역할 분리는 다음과 같다.

- `StationSnapshot`
  현재 검색 결과와 stale/fresh 판정의 입력
- `StationPriceHistory`
  직전 조회 대비 가격 변화 계산의 근거
- `WatchedStation`
  관심 비교 화면과 재방문 시나리오의 입력

가격 히스토리는 무한 저장하지 않는다. 주유소별 최근 성공 스냅샷 10개만 유지하는 trimming 정책을 둔다.

관심 주유소는 DataStore보다 Room 테이블로 관리한다. 이유는 아래와 같다.

- 주유소 식별자와 메타데이터를 함께 저장하기 쉽다.
- 마지막 확인 시각, 최근 비교값 등 조회 친화적 읽기 모델을 구성하기 쉽다.
- 추후 정렬, 조인, 비교 쿼리로 확장하기 쉽다.

## 상태 모델

기존 원칙인 "영속 선호 상태"와 "세션 상태"의 분리는 유지한다.

- 영속 선호 상태
  검색 반경, 유종, 브랜드 필터, 정렬 기준, 외부 지도 제공자
- 세션 상태
  현재 좌표, 권한 상태, GPS 상태, 로딩/새로고침 상태, 스낵바/외부 지도 효과
- 읽기 모델 상태
  스냅샷 결과, 가격 변화, 관심 여부, 마지막 확인 시각, 비교 화면 집계 결과

리스트 아이템은 아래 파생 필드를 포함하도록 확장한다.

- `priceDelta`
- `isWatched`
- `lastSeenAt`

비교 화면은 아래 입력을 합성해서 만든다.

- 관심 주유소 집합
- 현재 위치
- 최신 가격 스냅샷
- 가격 히스토리

이 설계는 화면 책임을 명확히 하고, 상태 테스트를 독립적으로 작성하기 쉽게 만든다.

## 데이터 흐름

대표 흐름은 아래와 같다.

1. 사용자가 리스트를 새로고침한다.
2. 현재 위치와 사용자 선호 설정으로 `StationQuery`를 만든다.
3. repository가 원격 데이터로 스냅샷을 갱신하고, 동시에 가격 히스토리를 기록한다.
4. 리스트 화면은 최신 스냅샷과 히스토리를 조합해 가격 변화 배지를 렌더링한다.
5. 사용자가 특정 주유소를 관심 등록한다.
6. 비교 화면은 관심 주유소 목록, 최신 스냅샷, 현재 위치, 히스토리를 조합해 읽기 모델을 만든다.

핵심은 "원본 저장"보다 "설명 가능한 읽기 모델"을 만드는 데 있다.

## 오류 처리와 예외 시나리오

기존 stale 데이터 유지 정책은 그대로 유지한다.

- 새로고침 실패 시 마지막 성공 스냅샷을 계속 렌더링한다.
- stale 상태에서는 마지막 갱신 시각과 stale 메시지를 명시한다.
- 가격 히스토리가 부족하면 가격 변화 배지는 `비교 데이터 없음`으로 처리한다.
- 관심 주유소가 현재 검색 결과에 없더라도, 저장된 항목은 비교 화면에서 마지막 확인 정보와 함께 유지한다.
- 위치 권한 거부, GPS 비활성화, 현재 위치 확인 실패 시 기존 effect 정책을 재사용한다.

이 설계는 실패 시 빈 화면보다 "마지막으로 신뢰 가능한 상태"를 유지하는 기존 철학을 확장 기능에도 일관되게 적용한다.

## 테스트 전략

검증 메시지는 "테스트를 많이 작성했다"가 아니라 "어떤 리스크를 어떤 층에서 통제했는가"여야 한다.

### 1. 도메인/리포지토리 테스트

아래 규칙을 테스트한다.

- 가격 변화 계산 규칙
- 히스토리 trimming 정책
- 관심 주유소 추가/제거/중복 방지
- stale snapshot과 비교 데이터 결합 규칙
- 관심 주유소가 현재 검색 결과에서 누락된 경우의 fallback 규칙

### 2. ViewModel 테스트

아래 상태 전이를 검증한다.

- 관심 등록/해제 후 UI state 반영
- 정렬 기준 변경 후 리스트와 비교 화면 state 갱신
- refresh 실패 시 마지막 성공 데이터 유지
- 위치 권한, GPS, 오프라인 조건에서 effect와 loading state 변화

### 3. UI 테스트

핵심 사용자 흐름을 계측 테스트로 검증한다.

- 리스트에서 관심 등록
- 비교 화면 진입
- 정렬/필터 변경
- stale 상태 메시지 및 가격 변화 배지 노출
- 외부 지도 액션 호출

## 성능 설계

현재 benchmark 모듈은 baseline profile 생성 기반만 갖고 있다. 포트폴리오용으로는 측정 시나리오를 아래까지 확장한다.

- cold start 시간
- 첫 리스트 렌더링 시간
- 정렬 전환 시 프레임 안정성
- 비교 화면 진입 시간

README에는 단순 숫자 나열이 아니라, 왜 이 시나리오를 측정했는지 설명한다.

- Compose 기반 리스트 진입이 첫인상 성능에 중요함
- 정렬 전환은 상태 결합과 리스트 재구성 비용을 드러냄
- 비교 화면 진입은 확장 기능의 복잡도를 보여줌

## 분석 이벤트 설계

분석 이벤트는 마케팅 목적이 아니라 제품 판단과 상태 추적 설계를 보여주기 위한 것이다.

기본 이벤트 세트는 다음과 같다.

- `station_search_refreshed`
- `station_filter_changed`
- `station_sort_toggled`
- `station_watch_toggled`
- `station_compare_viewed`
- `external_map_opened`

이벤트에는 최소한 아래 맥락을 함께 담는다.

- 검색 반경
- 유종
- 정렬 기준
- stale 여부
- 관심 주유소 여부
- 선택한 지도 제공자

필요 이상으로 무거운 analytics SDK 통합까지는 범위에 넣지 않는다. 우선은 이벤트 모델과 주입 가능한 추상화, 테스트 가능한 기록 구조를 우선한다.

## 포트폴리오 노출 설계

### README

README는 문서가 아니라 면접관용 랜딩 페이지처럼 읽혀야 한다. 구성은 아래 순서를 권장한다.

1. 한 줄 소개
2. 대표 스크린샷 3~4장
3. 핵심 기능
4. 아키텍처 요약 다이어그램
5. 상태 모델 / 오프라인 전략 요약
6. 테스트 및 benchmark 결과
7. 실행 방법
8. 기술 판단 포인트와 회고

### 스크린샷

단순 화면 캡처보다 의미 있는 상태를 보여주는 장면을 선택한다.

- 기본 리스트 화면
- stale 데이터 표시 상태
- 설정 변경 화면
- 관심 주유소 비교 화면
- 외부 지도 연동 직전 또는 직후 흐름

### 시연 영상

60초에서 90초 사이의 짧은 데모 영상을 만든다. 흐름은 아래 순서를 권장한다.

1. 앱 실행
2. 주변 주유소 조회
3. 정렬/필터 변경
4. 관심 주유소 저장
5. 비교 화면 확인
6. 외부 지도 열기
7. stale/offline 상태 짧게 시연

## 구현 범위 원칙

아래는 이번 설계에서 의도적으로 제외한다.

- 무거운 상세 페이지와 대규모 CRUD 확장
- 서버 백엔드 구축
- 복잡한 계정 기능
- 운영용 실시간 analytics 인프라
- 현재 핵심 가치와 무관한 대규모 리팩터링

범위를 통제하는 이유는 "대표 기능 1개를 끝까지 설계하고 검증한다"는 메시지를 지키기 위해서다.

## 성공 기준

이 설계가 완료되면 GasStation은 아래 네 문장으로 설명 가능해야 한다.

- 위치, 권한, 오프라인, 외부 앱 연동이 섞인 실무형 탐색 앱을 만들었다.
- 멀티모듈 구조로 책임을 분리하고 상태를 영속/세션/읽기 모델로 나눴다.
- 가격 비교와 관심 주유소 기능으로 단순 조회 앱을 넘어서는 제품 가치를 만들었다.
