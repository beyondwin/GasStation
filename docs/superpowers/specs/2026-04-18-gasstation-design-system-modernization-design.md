# GasStation Design System Modernization Design

작성일: 2026-04-18

## 1. 문서 목적

이 문서는 GasStation의 현재 `legacy` UI를 버리지 않고, `노랑 + 검정 + 흰 카드`라는 시각 정체성을 유지한 채 더 현대적인 Compose UI 시스템으로 재정비하기 위한 설계 문서다.

목표는 화면을 예쁘게 다시 칠하는 것이 아니다. 다음 세 가지를 동시에 달성하는 것이 목적이다.

- 모든 화면이 같은 안내 시스템 안에 있는 것처럼 느껴진다.
- 가격 중심 정보 위계가 흔들리지 않는다.
- 새 화면을 추가해도 토큰과 공통 컴포넌트 규칙만으로 일관성을 유지할 수 있다.

## 2. 설계 고정값

이번 설계에서 아래 항목은 열린 질문이 아니라 고정값으로 취급한다.

- 색상과 기본 브랜드 톤은 유지한다.
- 시각 방향은 `공공 안내판형` 현대화로 잡는다.
- 우선 개선 범위는 `공통 디자인 시스템`이다.
- 포함 범위는 `타이포그래피 + 간격/shape/stroke/icon 토큰 + 공통 컴포넌트 규칙`이다.
- 주유소 목록 화면에서 사용자가 가장 먼저 읽어야 하는 정보는 `가격`이다.
- 카드 밀도는 과밀도도 과여백도 아닌 `중간 균형형`으로 유지한다.

## 3. 디자인 방향

### 3.1 톤

새 방향은 “브랜드 앱”보다 “빠르게 믿고 읽을 수 있는 도시형 공공 안내판”에 가깝다.

지켜야 할 인상은 아래와 같다.

- 노랑/검정의 높은 대비는 유지한다.
- 강한 대비는 남기되, 텍스트와 컴포넌트는 더 차분하고 정돈된 리듬을 가진다.
- 배너, 필터, 카드, 설정 행이 서로 다른 제품처럼 보이지 않는다.
- 사용자가 앱을 닫고 나서 떠올리는 이미지는 “가격이 가장 먼저 읽히는 믿을 만한 주유 안내판”이다.

### 3.2 비시각 원칙

- 가격은 카드의 주인공이다.
- 나머지 정보는 가격을 보조해야 하며 경쟁하면 안 된다.
- 공통 컴포넌트는 자유도가 아니라 제약을 통해 일관성을 만든다.
- 공통 컴포넌트는 각자 허용된 텍스트 역할만 사용한다.

## 4. 현재 문제 정의

현재 `core:designsystem`과 각 feature 화면의 조합에는 다음 문제가 있다.

- 타이포그래피 역할 수가 부족하고 역할명이 실제 용도와 어긋나 있다.
- `labelLarge = 32sp`처럼 이름과 의미가 어긋난 스타일이 큰 숫자 표시 역할까지 겸하고 있다.
- 목록, 배너, 필터, 설정 행이 서로 다른 의미의 텍스트를 `title`/`label` 슬롯에 임의로 얹고 있다.
- 공통 컴포넌트는 이미 존재하지만, 그 내부 정보 위계를 강제하는 시스템은 없다.
- 결과적으로 “왜 이 텍스트가 이 크기인가”가 설명되지 않는 상태다.

이번 설계는 이 문제를 “타이포 값 조정”이 아니라 “의미 기반 시스템 재정의”로 푼다.

## 5. 시스템 구조

### 5.1 기본 원칙

앱 내부 화면은 `MaterialTheme.typography.titleMedium` 같은 Material 기본 슬롯을 직접 고르는 방식에서 벗어난다.

대신 `core:designsystem`에 앱 의미 기준 토큰 계층을 두고, feature 화면과 공통 컴포넌트는 그 토큰만 소비한다.

Material typography는 아래 역할로 제한한다.

- 외부 Material 컴포넌트와의 호환
- 앱 토큰 구현의 기반 값 제공
- Preview와 테스트에서의 기본 fallback

앱 화면의 직접 표현 의미는 Material 슬롯명이 아니라 앱 토큰이 소유한다.

### 5.2 토큰 계층

정비 대상 토큰은 아래와 같다.

- `Typography`
- `Spacing`
- `Corner`
- `Stroke`
- `IconSize`
- 필요 시 얕은 `Elevation/Shadow`

토큰은 “크다/작다”가 아니라 “무슨 역할인가”로 정의한다.

## 6. 타이포그래피 설계

### 6.1 역할 기반 텍스트 스타일

새 타이포 계층은 아래 역할을 가진다.

- `TopBarTitle`
  - 상단 바 제목 전용
  - 현재 정렬 상태나 화면 제목처럼 상단 맥락을 대표하는 텍스트
- `SectionTitle`
  - `현재 조건`, `관심 비교`, 설정 섹션 제목 같은 구역 제목
- `CardTitle`
  - 주유소명, 주요 리스트 항목 제목
- `PriceHero`
  - 목록 카드에서 가장 먼저 읽히는 가격 숫자
  - 앱 내에서 가장 강한 숫자 강조 스타일
- `MetricValue`
  - 거리, 변동폭, 비교 화면 보조 수치
  - `PriceHero`보다 한 단계 낮은 숫자 스타일
- `Body`
  - 상태 설명, 안내 문구, 일반 본문
- `Meta`
  - 브랜드, 갱신 시각, 보조 정보, 단위
- `Chip`
  - 반경, 유종, 필터 pill 내부 텍스트
- `BannerTitle`
  - 배너 핵심 상태 문구
- `BannerBody`
  - 배너 보조 설명

### 6.2 강제 규칙

- 큰 숫자 스타일은 `PriceHero`와 `MetricValue`만 사용한다.
- `Chip`, `Banner`, 설정 행 보조 정보에는 큰 숫자 스타일을 금지한다.
- 단위 표시는 수치보다 약해야 하며, 숫자와 시각 경쟁을 하면 안 된다.
- 가격 이외의 텍스트는 가격보다 굵거나 크게 보이지 않는다.
- 리스트 항목명은 `CardTitle`을 사용하되 가격보다 앞서 보이지 않는다.

## 7. 비타이포 토큰 설계

### 7.1 Spacing

리듬은 과하게 세분화하지 않고 아래 축을 기준으로 단순화한다.

- `4`
- `8`
- `12`
- `16`
- `24`

이 값은 카드 내부, 배너 내부, 설정 행, top bar 액션 간격에 공통으로 적용한다. 같은 깊이의 요소는 같은 간격 리듬을 가져야 한다.

### 7.2 Corner

모서리 반경은 역할별 3단계 정도로 제한한다.

- 소형: chip, 작은 보조 표면
- 중형: 배너, 설정 행, 내부 패널
- 대형: 주요 카드, 상위 chrome 표면

현재처럼 모든 요소가 서로 다른 반경을 갖는 상태는 지양한다.

### 7.3 Stroke

브랜드 특유의 검정 외곽선은 유지하되 두께 레벨은 제한한다.

- 기본 외곽선
- 강조 외곽선

필터, 카드, 배너가 각자 다른 선 두께로 떠들지 않게 한다.

### 7.4 IconSize

아이콘 크기는 아래 용도로 분리한다.

- top bar action
- trailing action
- 상태/선택 아이콘

텍스트 크기에 따라 아이콘이 들쑥날쑥 커지는 일을 막는다.

## 8. 공통 컴포넌트 규칙

### 8.1 TopBar

`TopBar`는 시각 정체성을 유지하되 본문보다 먼저 과하게 튀지 않도록 내부 밀도를 조정한다.

규칙:

- 제목은 `TopBarTitle`만 사용한다.
- 텍스트 액션과 아이콘 액션을 혼용해도 높이와 패딩은 고정한다.
- 액션은 브랜드 대비를 유지하되 본문 카드의 정보 주도권을 침범하지 않는다.

### 8.2 ChromeCard

`ChromeCard`는 공통 카드 프레임이며, 내부 정보 구조도 같이 강제한다.

허용 슬롯:

- `Header`
- `Primary Metric`
- `Supporting Info`
- `Actions`

카드마다 임의의 위계를 만들지 않고, 공통 카드가 정보를 정리하는 역할을 가져야 한다.

### 8.3 MetricBlock

`MetricBlock`은 수치 정보 전용 블록이다.

규칙:

- 가격은 `PriceHero`를 사용한다.
- 거리/변동폭은 `MetricValue`를 사용한다.
- 단위는 항상 보조 레벨로 처리한다.
- 수치 블록은 읽기 속도를 높여야 하며 장식이 되어서는 안 된다.

### 8.4 StatusBanner

`StatusBanner`는 화면 상태를 전달하는 안내 표면이다.

규칙:

- `BannerTitle + BannerBody` 2단 구조를 기본으로 한다.
- 카드와 비슷한 위계를 차지하지 않는다.
- stale/offline/refresh/error 상태는 모두 같은 안내 계열 안에서 표현한다.

### 8.5 FilterPill / FuelChip

필터와 칩은 맥락 표시 컴포넌트다.

규칙:

- 큰 글자 금지
- 과한 외곽선 중첩 금지
- 본문보다 먼저 읽히지 않도록 조용한 위계 유지

### 8.6 ListRow / SettingsRow

설정 행과 일반 보조 리스트 행은 같은 구조적 문법을 공유한다.

권장 구조:

- `overline`
- `title`
- `subtitle`
- `meta`

각 슬롯의 사용 여부는 명확히 정의하고, feature 화면이 임의로 역할을 뒤섞지 않도록 한다.

## 9. 화면 적용 전략

적용 순서는 아래가 기준이다.

1. `core:designsystem`
2. `feature:station-list`
3. `feature:watchlist`
4. `feature:settings`

이 순서를 택하는 이유는 `station-list`가 새 시스템의 기준 화면이기 때문이다. 목록 화면은 가격 우선 위계, 카드 밀도, 필터, 배너, 액션 균형을 모두 동시에 검증할 수 있다.

### 9.1 주유소 목록

목록 화면은 새 시스템의 기준 화면이다.

원칙:

- 가격이 가장 먼저 읽힌다.
- 주유소명은 가격을 설명하는 제목으로 동작한다.
- 거리, 변동폭, 브랜드, 갱신 시각은 2차 정보다.
- 필터 pill과 상태 배너는 본문과 경쟁하지 않는다.

### 9.2 관심 비교

관심 비교 화면은 동일한 토큰을 사용하되, 한 카드의 강한 위계보다 여러 카드의 비교 스캔성이 더 중요하다.

원칙:

- 가격은 여전히 핵심 정보다.
- 카드 간 수치 정렬성이 중요하다.
- 목록 화면과 다른 제품처럼 보이지 않는다.

### 9.3 설정

설정 화면은 장식보다 안정감이 중요하다.

원칙:

- 섹션 제목, 옵션 제목, 보조 설명, 현재 선택값 위계를 명확히 구분한다.
- 목록/배너/카드와 같은 시스템 안에 있어야 한다.
- 설정 화면이 별도 UI 섬처럼 보이지 않도록 한다.

## 10. 상태 화면 설계 원칙

대상 상태:

- 권한 필요
- GPS 꺼짐
- 로딩
- 빈 결과
- stale/offline

원칙:

- 각각을 별도 제품 화면처럼 만들지 않는다.
- 같은 안내 시스템 안의 상태 변화처럼 보여야 한다.
- `stale/offline`은 위험 경고보다 신뢰도 보정 정보에 가깝게 다룬다.
- 로딩에서도 가능하면 화면 맥락을 잃지 않도록 제목/조건 영역을 유지한다.

## 11. 검증 기준

### 11.1 디자인 시스템 검증

- 큰 숫자 스타일이 `PriceHero`/`MetricValue` 바깥으로 새지 않는다.
- 공통 컴포넌트가 허용된 텍스트 역할만 사용한다.
- 간격, 반경, 외곽선, 아이콘 크기가 역할 단위로 반복 가능하다.

### 11.2 화면 검증

- 목록 화면에서 가격이 가장 먼저 읽힌다.
- 관심 비교 화면에서 여러 카드의 수치 정렬이 흐트러지지 않는다.
- 설정 화면의 세로 리듬이 일정하다.
- 배너와 필터가 본문과 경쟁하지 않는다.

### 11.3 회귀 방지

- 토큰 단위 테스트를 추가한다.
- 공통 컴포넌트 테스트로 허용 역할을 검증한다.
- semantics 기반 위치/정렬 테스트 또는 스크린샷 테스트를 일부 도입한다.

## 12. 범위와 비범위

### 12.1 범위

- `core:designsystem`의 타이포/토큰 체계 재정의
- 공통 컴포넌트 텍스트 역할 정리
- 목록/관심 비교/설정 화면의 새 시스템 적용 기준 수립
- 상태 화면의 안내 시스템 정렬

### 12.2 비범위

- 도메인 모델과 저장소 계약 변경
- 색상 정체성 자체의 변경
- 완전히 새로운 브랜드 스타일 도입
- 기능 의미나 사용자 흐름의 대규모 변경

## 13. 완료 기준

이번 재설계는 아래가 충족되면 완료로 본다.

- 모든 화면이 같은 안내 시스템처럼 느껴진다.
- 가격 중심 위계가 흔들리지 않는다.
- 큰 글자와 작은 글자의 역할이 명확하다.
- 상태 배너, 필터, 설정 행이 본문과 경쟁하지 않는다.
- 새 화면을 추가할 때도 공통 토큰과 컴포넌트 규칙만으로 일관성을 유지할 수 있다.
