# GasStation 설계 문서 검토 게이트

작성일: 2026-04-18  
대상 문서: `/Users/kws/.gstack/projects/beyondwin-GasStation/kws-main-design-20260418-033029.md`  
목적: 상세 구현 문서를 작성하기 전에, 현재 설계 문서의 방향성과 설계 구멍을 검토하고 보강 포인트를 확정한다.

## 1. 이 문서의 역할

이 문서는 구현 계획서가 아니다.  
또, 기존 설계 문서를 처음부터 다시 쓰는 문서도 아니다.

역할은 하나다.

현재 설계 문서가:
- 방향이 맞는지
- 구현 전에 위험한 구멍이 무엇인지
- 어떤 결정을 먼저 고정해야 상세 구현 문서로 내려갈 수 있는지

를 검토하고, 설계 문서를 더 강한 상태로 개정하기 위한 기준을 만드는 것이다.

## 2. 현재 설계 문서가 잘 잡고 있는 방향

현재 설계 문서의 큰 방향은 맞다.

특히 아래 판단은 유지해야 한다.
- 이 프로젝트는 “예쁜 주유소 앱”이 아니라 `면접용 레퍼런스 앱`이다.
- 1단계는 `C`, 즉 `아키텍처 + Flow`를 완성하는 단계다.
- 2단계는 `D`, 즉 오프라인, 품질 자동화, 성능, 배포까지 얹는 단계다.
- 현재 레거시 앱은 구조 템플릿이 아니라 기능 요구사항 참고자료다.

이 네 가지가 흔들리면 문서는 다시 “깔끔한 리팩터링 계획”으로 약해진다.

## 3. 현재 설계 문서의 핵심 구멍

### 3.1 캐시 정책이 선언형 수준에 머물러 있다

현재 문서는 `Room`을 단일 기준 원천으로 둔다고 했지만, 위치 기반 검색 앱에서 더 중요한 것은 캐시 규칙이다.

아직 빠져 있는 결정:
- 캐시 키를 무엇으로 구성할지
- `sortOrder`를 캐시 키에 넣을지 말지
- 브랜드 필터를 서버 조건으로 볼지 클라이언트 필터로 볼지
- 오래된 데이터 기준을 몇 분으로 볼지
- 사용자가 조금 이동했을 때 재조회 기준이 무엇인지

이게 없으면 `오프라인 우선`이 아니라 “Room도 넣은 앱”이 된다.

### 3.2 위치 권한 전략이 도메인 특성과 완전히 정렬되어 있지 않다

현재 문서는 `coarse-first` 논리를 포함하고 있는데, 이 앱은 근거리 검색 정확도가 핵심이다.

문제:
- 3km 반경 검색과 거리 정렬은 위치 정확도에 민감하다
- 외부 지도 앱 연동은 대략적 위치만 허용된 경우에도 동작은 하지만 품질이 떨어질 수 있다

따라서 기본 정책은 `coarse-first`보다 `정밀 위치 우선, 대략 위치 허용` 정책이 더 적합하다.

### 3.3 설정의 소유권이 불명확하다

현재 문서는 `DataStore`를 쓰겠다고 했지만, 어떤 값이 “장기 사용자 선호”이고 어떤 값이 “현재 화면 세션 상태”인지 분리가 덜 되어 있다.

이게 불명확하면:
- `feature:settings`와 `feature:station-list` 경계가 흐려지고
- 홈 화면이 다시 거대한 상태 홀더가 되며
- 설정 변경이 즉시 저장되는지, 현재 세션에만 적용되는지 해석이 갈린다

### 3.4 모델 계층 규칙이 약하다

`DTO와 도메인 모델을 분리한다`는 선언은 있지만, 그 정도로는 부족하다.

지금 상태로 구현에 들어가면 `core:model`이 모든 도메인 모델을 먹는 공용 창고가 될 위험이 크다.

반드시 명시해야 할 계층:
- 원격 DTO
- 로컬 엔티티
- 도메인 모델
- UI 모델

또한 매퍼 소유권도 같이 정해야 한다.

### 3.5 검토자 온보딩 전략이 아직 선택사항처럼 보인다

현재 문서에서는 `demo flavor`가 열린 질문에 가깝다.

하지만 이 프로젝트는 포트폴리오용 레포다.  
API 키 없이도 돌려볼 수 있는 기본 실행 경로가 없으면, 구조가 좋아도 레포는 약해진다.

`demo`는 선택이 아니라 거의 필수다.

### 3.6 마이그레이션의 절단면이 약하다

현재 문서는 `1차 패스`, `2차 패스`는 잘 잡았지만, 레거시에서 새 구조로 넘어갈 때 무엇을 어떤 단위로 끊어 구현할지는 아직 약하다.

기능 목록 기준으로 자르면 다시 구조가 흔들릴 가능성이 크다.  
상태와 계약 단위로 마이그레이션 절단면을 정의해야 한다.

## 4. 상세 구현 문서 전에 반드시 고정해야 하는 결정

아래 항목은 구현 문서로 미루면 안 된다.

### 4.1 캐시 정책
- 캐시 키 구성 규칙
- 오래된 데이터 기준 시간
- location bucket 기준
- 서버 조건과 클라이언트 필터의 분리 기준

### 4.2 위치 권한 정책
- 기본 권한 경로를 `정밀 위치 우선`으로 둘지
- 대략적 위치 허용 시 성능 저하 모드를 어떻게 표현할지

### 4.3 설정 소유권 규칙
- 영속 선호값과 세션 상태의 분리
- 홈 화면 액션이 언제 DataStore에 즉시 반영되는지

### 4.4 모델 계층 규칙
- DTO / Entity / Domain / UiModel 분리
- `core:model` 범위 축소
- 매퍼 소유권 명시

### 4.5 실행 모드 전략
- `demo` 기본 제공 여부
- `prod` 활성화 조건
- 가짜 위치 / 픽스처 데이터 전략

### 4.6 마이그레이션 순서와 완료 기준
- 어떤 계약부터 자를지
- 동등성을 어떤 기준으로 판단할지
- 단계 완료를 기능이 아니라 계약/상태 기준으로 볼지

## 5. 현재 문서에 추가해야 할 신규 섹션

현재 설계 문서는 `핵심 설계 원칙` 다음에 바로 `목표 모듈 그래프`로 넘어간다.

이 사이에 아래 상위 섹션을 새로 넣어야 한다.

## 설계 고정값

이 상위 섹션 아래에 다음 6개를 배치한다.

### 5.1 캐시 정책

권장 고정 문장:
- 캐시는 화면 재구성을 위한 읽기 모델로 취급한다.
- 캐시 키는 `locationBucket + searchRadius + fuelType`으로 구성한다.
- `sortOrder`와 `mapProvider`는 캐시 키에서 제외한다.
- 브랜드 필터는 가능한 경우 클라이언트 필터로 처리한다.
- 오래된 데이터도 제품 상태로 인정하며, 마지막 성공 결과는 실패 시에도 유지한다.

### 5.2 위치 권한 정책

권장 고정 문장:
- 이 앱의 기본 권한 경로는 `정밀 위치 우선, 대략 위치 허용`이다.
- 대략적 위치만 허용된 경우에도 검색 기능은 성능 저하 모드로 동작해야 한다.
- 권한 요청은 문맥 안에서 수행한다.
- GPS 비활성, 권한 거부, 대략적 위치만 허용된 상태는 별도 UI 상태로 표현한다.

### 5.3 설정 소유권 규칙

권장 고정 문장:
- `DataStore`는 장기 사용자 선호값의 단일 기준 원천이다.
- 현재 위치, 권한 상태, 오래된 데이터, 새로고침, 스낵바, 지도 실행 효과는 세션 상태로만 관리한다.
- 영속 선호값과 세션 상태는 다른 저장 수명과 책임을 가진다.

### 5.4 모델 계층 규칙

권장 고정 문장:
- 원격 DTO, 로컬 엔티티, 도메인 모델, UI 모델을 명시적으로 분리한다.
- `core:model`에는 기능에 종속되지 않는 공용 값 객체만 둔다.
- 기능별 도메인 모델은 각 `domain:*` 모듈이 소유한다.
- 매퍼 소유권은 `data`와 `feature/presentation` 계층에 명시적으로 배정한다.

### 5.5 실행 모드 전략

권장 고정 문장:
- `demo`는 검토자 온보딩을 위한 기본 실행 경로다.
- `demo`는 API 키 없이 실행 가능해야 한다.
- `prod`는 로컬 시크릿이 있을 때만 활성화한다.
- `demo`는 픽스처 데이터와 가짜 위치 제공자를 사용해 대표 상태를 재현할 수 있어야 한다.

### 5.6 마이그레이션 순서와 완료 기준

권장 고정 문장:
- 재작성은 화면 단위보다 계약/상태 단위로 진행한다.
- 우선순위는 `settings contract -> location contract -> station query pipeline -> station list presentation -> external map handoff` 순으로 둔다.
- 단계 완료 기준은 “동작함”이 아니라 “소유 모듈, 상태 모델, 계약, 테스트 전략이 정의됨”이다.

## 6. 현재 문서에서 수정해야 할 기존 섹션

### 6.1 `목표 모듈 그래프`

현재 문장:
- 모듈은 파일 종류 기준이 아니라 제품 책임 기준으로 나눈다.

개정 방향:
- 앞서 정의한 캐시 정책, 설정 소유권, 모델 계층 규칙을 반영한 결과물이라는 설명을 추가한다.

### 6.2 `모듈별 역할`

특히 `core:model` 설명을 축소해야 한다.

수정 방향:
- `core:model`은 공용 값 객체만 소유한다.
- 주유소/검색조건/설정 같은 도메인 모델은 `domain:*` 모듈 내부로 이동한다.

### 6.3 `UI와 상태 모델`

수정 방향:
- 영속 선호값은 입력 상태이고
- 현재 위치, 권한, 오래된 데이터, 새로고침, 효과는 세션 상태라는 점을 연결 문장으로 추가한다.

### 6.4 `오프라인 우선 전략`

수정 방향:
- 캐시 키, 신선도, 오래된 데이터 규칙 같은 결정문은 `설계 고정값 > 캐시 정책`으로 이동한다.
- 이 섹션은 왜 Room을 단일 기준 원천으로 쓰는지와 범위 선언만 남긴다.

### 6.5 `위치 권한과 위치 조회 설계`

수정 방향:
- `coarse-first` 논리를 제거한다.
- `정밀 위치 우선, 대략 위치 허용` 정책을 기반으로 설명을 다시 쓴다.

### 6.6 `열린 질문`

수정 방향:
- 아래 항목은 열린 질문에서 제거한다.
  - demo flavor 여부
  - 위치 권한 기본 경로
  - DataStore 소유권 규칙 여부
- 아래처럼 수치와 전략의 미세 조정만 남긴다.
  - location bucket 250m vs 300m
  - 오래된 데이터 기준 5분 vs 10분
  - 브랜드 필터를 전부 클라이언트 측에서 둘지 여부

## 7. 권장 문서 흐름

개정 후 문서는 아래 순서로 읽혀야 한다.

1. 왜 이 앱을 다시 만드는가
2. 어떤 기준으로 좋은 레퍼런스 앱을 만들 것인가
3. 무엇이 이미 결정되었는가
4. 그 결정 때문에 모듈이 어떻게 나뉘는가
5. 그 구조 안에서 상태와 데이터가 어떻게 흐르는가
6. 어떤 순서로 구현할 것인가
7. 언제 완료로 볼 것인가

현재 문서는 1, 2, 4, 5, 6은 강하다.  
가장 약한 부분은 3이다.  
이번 개정의 목적은 정확히 그 빈칸을 메우는 것이다.

## 8. 이 검토 문서를 기준으로 다음에 해야 할 일

상세 구현 문서로 내려가기 전에 현재 설계 문서에 아래 작업을 반영한다.

1. `설계 고정값` 섹션 신설
2. `core:model` 책임 축소
3. `coarse-first` 제거
4. `demo` 실행 경로를 열린 질문이 아니라 고정 전략으로 승격
5. 마이그레이션 기준을 계약/상태 중심으로 재정의
6. 열린 질문을 수치 조정 수준으로만 축소

이 여섯 개가 반영되면, 현재 설계 문서는 “좋은 방향의 초안”에서 “상세 구현 문서 작성 전 검토를 통과할 수 있는 설계 문서”로 올라간다.

## 9. 스펙 자체 점검

이 문서는 아래 기준으로 자체 점검을 통과해야 한다.

- 플레이스홀더 없음
- 현재 설계 문서의 목적과 충돌하지 않음
- 새로운 섹션 추가가 아니라, 현재 문서의 약한 축을 보강하는 방향임
- 구현 디테일을 과도하게 선점하지 않음
- 다음 단계가 “상세 구현 문서 작성”으로 자연스럽게 이어짐
